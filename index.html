01Why rez exists

You're a rigger or TA. You've hit at least one of these:

  • Show A is on Maya 2022 + Python 3.7. Show B needs Maya 2024 + Python 3.10. Both shows are active, and the lighter wants you to fix something on Show A while you're mid-sprint on Show B.
  • Your rig tool needs PySide2. The new pipeline tool needs PySide6. Loading both into Maya at once causes a hard crash.
  • You wrote a script that works on your machine. It crashes on the modeler's machine because their maya.env has a different MAYA_MODULE_PATH.
  • The pipeline TD wants to upgrade everyone's Python, but five tools haven't been ported yet, and rolling back is "complicated."

The traditional studio workaround is a swarm of .bat files that set environment variables before launching Maya, plus a maze of maya.env overrides. It works until it doesn't, and when it doesn't, you debug by reading shell scripts at 2 AM.

Rez replaces all of that with one idea: declare what your tool needs, and rez figures out the environment.

Two artists. Two shows. Two completely different software stacks. Rez gives each subshell its own resolve.

02The one concept: a package

If you understand one thing about rez, understand this: a package is a folder that says "I am a thing. Here's what I depend on. Here's what to do to the environment when I'm active."

On disk, every package looks the same:

The convention is rigid: <packages_path>/<name>/<version>/package.py. Multiple versions of the same package live side by side.

Here's the smallest useful package.py:

name = "testRez"
version = "0.1.0"

requires = [
    "python-3.8",
]

def commands():
    env.MY_PROJECT_ROOT = "{root}"
    env.PYTHONPATH.append("{root}/python")

Three things to notice:

  1. requires is a list of "what other packages I need." python-3.8 is a version range — any 3.8.x will satisfy it.
  2. commands() describes how the environment should change when this package is part of a resolve. Set env vars, append to PYTHONPATH, define aliases — same kinds of things you'd put in a .bat launcher, but declarative.
  3. {root} is a placeholder rez fills in with the absolute path to this package on disk at resolve time. So if your tool's source code lives at {root}/python/, rez puts that on PYTHONPATH automatically.
TA framing
Think of package.py as the thing you'd hand to a new artist on day one — "here is the recipe for the environment my tool needs." Except instead of a wiki page, the recipe is executable.

04commands() and the sandbox idea

This is the part newcomers usually trip on. commands() looks like a normal Python function, but it doesn't actually do anything when it runs. It records intentions, which rez later transpiles into your shell's native script.

The same commands() body produces different shell-native output depending on the target shell. Riggers can write platform-agnostic recipes; rez translates.

The vocabulary inside commands() is small:

You writeIt does
env.X = "value"Set env var X
env.PATH.prepend("...")Prepend to PATH (also append)
env.PYTHONPATH.append("...")Same idea for any list-style var
alias("maya", "maya.exe")Define a shell alias / function
info("message")Print a banner when the env loads
if "maya-2024" in resolve:Branch on what else is in the resolve
{root}Token for "this package's install dir"
Common gotcha
You can't use arbitrary Python in commands() — file I/O, network calls, os module access, etc. won't work as you'd expect. The body is executed in a sandbox where only the rex API is "live." Treat it as a config DSL that happens to look like Python, not a normal script.

05Variants: one package, many platforms

A single package version can ship multiple variants — typically one per OS or per supported dependency version. Inside the version folder, each variant gets its own subdirectory:

The variants list in package.py tells rez "I ship pre-built copies for these specific combinations." The solver picks the matching one.
name = "rigKit"
version = "1.4"
variants = [
    ["platform-windows", "arch-AMD64", "python-3.10"],
    ["platform-linux",   "arch-x86_64", "python-3.10"],
]

The big payoff: {root} inside commands() resolves to the variant's subdirectory, not the version's root. So your PYTHONPATH.append("{root}/python") magically points at the right per-platform binaries with no if statements.

06The studio go pattern

Most studios wrap rez-env in a one-liner shell function so artists never type rez-env directly. The convention is go <project>:

# PowerShell profile
function go {
    param(
        [Parameter(Mandatory=$true, Position=0)] [string]$Project,
        [Parameter(ValueFromRemainingArguments=$true)] [string[]]$Extra
    )
    & rez-env $Project @Extra
}

That's the entire mechanism. go showA runs rez-env showA. go showA maya-2024 runs rez-env showA maya-2024 — handy when you want to swap one component on the fly.

What this gives an artist
"To work on Show A, I open a terminal and type go showA. To work on Show B, go showB. I never edit a maya.env. I never set a PATH variable. The launcher gets the right Maya, the right Python, the right plugins, every time."

07A real TA example

Let's make this concrete. You wrote a rig auto-build script. It's a Python package living at C:\dev\autoRig\. You want any artist on Show A to be able to run it inside Maya 2024.

Step 1 — make autoRig a rez package

Add a package.py at the repo root:

# C:\dev\autoRig\package.py
name = "autoRig"
version = "0.1.0"

requires = [
    "python-3.10",
    "PySide2",
]

def commands():
    env.PYTHONPATH.append("{root}/python")
    env.MAYA_SCRIPT_PATH.append("{root}/mel")
    env.MAYA_PLUG_IN_PATH.append("{root}/plugins")

Now {root}/python goes on PYTHONPATH in any resolve that includes autoRig. Maya picks up your scripts and plugins automatically.

Step 2 — install it into the local packages dir

cd C:\dev\autoRig
rez-build --install

This copies your code into C:\dev\rez-packages\autoRig\0.1.0\ (a "release" of v0.1.0). Now any other rez package can depend on it.

Step 3 — make showA depend on it

The Show A package definition adds autoRig to its requires list:

# C:\dev\rez-packages\showA\1.0.0\package.py
name = "showA"
version = "1.0.0"
requires = [
    "maya-2024",
    "python-3.10",
    "autoRig-0.1",
]

Step 4 — artist workflow

An artist on Show A opens a terminal and types:

go showA
maya

Maya launches with autoRig already on every relevant path. No env files, no copy-paste setup, no "did you remember to source the show?". When you ship autoRig-0.2.0, Show A just bumps the version in its requires and every artist gets it on their next go showA.

09Where to go next

  • Releases. rez-release is rez-build --install with version-bump and tag conventions, plus a release-target path that's usually a network share. This is how a studio publishes a new version of autoRig for everyone.
  • rez-pip. Wraps pip so any PyPI package becomes a rez package. rez-pip --install requests gives you a requests rez package.
  • Hooks. Run code at resolve time, build time, or release time. Useful for licensing checks, logging, or auto-incrementing version stamps.
  • Package overrides. A user can stick a ~/packages/showA/1.0.0-dev/package.py in their personal packages_path and rez will prefer the dev version over the released one. Great for "try my fix" workflows.
  • Memcached / cache backends. For large studios, the per-resolve cost matters. Rez has pluggable caches.

The official docs are the canonical reference once you're past the basics. The concepts above don't change; everything else is variations on these primitives.
